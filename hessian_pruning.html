How is H[i,i] computed for each weight?
For each weight wᵢ we ask: "If I nudge wᵢ by a tiny ε, how sharply does the loss curve?" That sharpness is the second derivative — the Hessian diagonal entry H[i,i]. We approximate it with the central difference formula using just 3 forward passes:
H[i,i] ≈ ( L(w+ε) − 2· L(w) + L(w−ε) ) / ε²     where ε = 0.001
This is repeated independently for every weight — 25 weights = 25 pairs of forward passes. Select any weight below, then step through the computation one stage at a time.

Step-by-Step Computation
1
Start at trained weights
The network is trained. We pick wᵢ — its current value is our center point for the curvature estimate.
2
Compute base loss L(w)
Run a full forward pass with all weights unchanged. Record this as our reference loss L(w).
3
Perturb wᵢ → wᵢ + ε
Increase only wᵢ by ε = 0.001. Run forward pass again. Record L(w⁺). All other weights stay fixed.
4
Perturb wᵢ → wᵢ − ε
Decrease only wᵢ by ε. Run forward pass. Record L(w⁻). Now we have both sides of the curvature.
5
Apply central difference formula
H[i,i] = (L⁺ − 2·L₀ + L⁻) / ε². This is the curvature — how "bowl-shaped" the loss is around wᵢ.
6
Compute OBD saliency score
Score = ½ · wᵢ² · H[i,i]. This is the predicted loss increase if we zero out wᵢ (prune it).

Pruning Workflow — Step by Step
Pruning is not just "delete small weights". Each method has a distinct workflow, and whether you need to retrain depends on the method and the sparsity level. Click any step below to see what it involves and why it matters.
① Magnitude Pruning Workflow

Simple, fast, and widely used. Treats weight size as a proxy for importance — but it's just a proxy.

1
Train network to convergence
Run standard gradient descent until loss is low. The weights settle into a trained configuration. Same for all methods.
2
Rank weights by |wᵢ| O(N log N)
Sort all weights by absolute value. Smallest = least important (assumption). No extra computation needed — you already have the weights.
3
Zero out bottom-k% weights destructive
Set the k% smallest weights to exactly 0. This is irreversible in one-shot pruning. The network immediately loses capacity — loss spikes.
4
Retrain (fine-tune) almost always required
The surviving weights need to re-adapt. Run more gradient descent epochs on the pruned network. Without this, accuracy collapses at >30% sparsity.
5
Evaluate & iterate iterative only
For best results: prune a little → retrain → prune more → retrain. Repeat until target sparsity is reached. This is iterative magnitude pruning.
② Hessian Pruning (OBD) Workflow

Uses second-order information. More expensive to compute, but can prune in one shot without retraining.

1
Train network to convergence
Same as magnitude. The Hessian approximation is most accurate at a minimum of the loss — so you must train fully before computing it.
2
Compute H[i,i] for each weight 3 × N forward passes
For each weight, run L(w+ε), L(w), L(w−ε). The curvature is (L⁺ − 2L₀ + L⁻)/ε². This is what Tab ③ shows step by step.
3
Score = ½ · wᵢ² · H[i,i] OBD formula
This is the predicted loss increase from removing wᵢ — not a proxy, but a mathematical estimate grounded in the loss landscape's curvature.
4
Zero out lowest-score weights mathematically safe
Remove weights with the smallest predicted impact. Because we chose carefully, the actual loss increase is small and bounded — especially at low-to-medium sparsity.
5
Retrain (optional) only at high sparsity
At low-to-medium sparsity (<50%), no retraining needed — this is the key advantage. At high sparsity (>50%), fine-tuning helps because the quadratic approximation drifts.
6
OBS upgrade: adjust remaining weights Optimal Brain Surgeon
Instead of retraining, OBS mathematically updates the remaining weights: Δw = −(wᵢ/H⁻¹[i,i]) · H⁻¹[:,i]. Free compensation — no gradient steps needed.

Quantization — Reducing Numerical Precision
Instead of zeroing weights (pruning), quantization rounds them to a coarser grid. float32 uses 32 bits per weight. int8 uses 8 bits. int4 uses 4 bits — 8× smaller than float32. The Hessian tells us which rounding errors are most costly — this is exactly how GPTQ quantizes LLMs like LLaMA and GPT.

GPTQ — Hessian-Weighted Quantization

Standard quantization rounds every weight equally. GPTQ weights the rounding error by the Hessian — weights in high-curvature regions get more precise rounding.

Standard error:  (w − round(w))²
GPTQ error:      (w − round(w))² · H[i,i]

Weights with high H[i,i] (steep curvature) need precise rounding. Weights with low H[i,i] (flat) can tolerate coarse rounding.

Each dot = one weight. X = rounding error magnitude. Y = H[i,i]. Top-right = most costly errors (GPTQ targets these). Bottom-left = cheap errors.

Knowledge Distillation — Teaching a Small Network
Instead of compressing one network, we train a tiny student network to imitate a larger teacher. The teacher's soft outputs (e.g. 0.87 instead of 1) carry richer information than hard labels — they encode the teacher's confidence and uncertainty. The student learns faster and generalises better by learning from these soft targets.

Why Soft Labels Work Better — The Intuition
Hard Label Training
Every point gets label ±1. A point right on the boundary gets the same label as a point deep inside a class. The student gets no information about uncertainty.
Soft Label Training
A boundary point gets teacher output 0.12 instead of 1. The student learns: "this point is ambiguous". This encodes the teacher's learned geometry of the problem.
Result
The student (7 params) trained with distillation approaches the teacher's decision boundary (25 params). More accurate than the same student trained on hard labels alone.
KV-Cache Optimization in LLMs

Every token a language model generates must attend to all previous tokens. Without caching, that means recomputing Key and Value matrices from scratch for every step — O(n²) work. The KV-Cache stores those intermediate tensors so each new token only does O(n) work. This tab shows exactly how that works, where the memory goes, and how MHA/GQA/MQA trade memory for speed.

① How KV-Cache Works — Token by Token

Each generated token computes Q, K, V from its embedding. Without cache every step recomputes K and V for all past tokens. With cache only the new token's K and V are computed — the rest are read from cache. Step through to see the difference.

② KV-Cache Memory Growth vs Sequence Length

Cache size = 2 × num_layers × num_heads × head_dim × seq_len × bytes_per_element. Drag the slider to see how quickly memory explodes for long contexts.

③ Multi-Head vs Grouped-Query vs Multi-Query Attention

The KV-Cache bottleneck inspired architectural variants that share K/V heads across multiple Q heads. GQA (used in Llama 3, Gemma 2) is the sweet spot: it cuts KV cache by G× with minimal accuracy loss.

④ PagedAttention — vLLM's KV-Cache Memory Manager

Traditional KV caches pre-allocate a contiguous block for the maximum sequence length — wasting GPU memory on tokens that haven't been generated yet. PagedAttention (vLLM) manages the cache like OS virtual memory: fixed-size pages are allocated on demand and can be scattered across physical memory.
